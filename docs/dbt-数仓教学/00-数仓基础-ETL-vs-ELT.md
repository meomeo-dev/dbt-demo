# 00 · 数仓与 ELT 基础

> 本章不含代码,是理解后面所有步骤的地基。读完你会明白:数据为什么要分层,dbt 为什么存在。

## ETL vs ELT:一字之差,天翻地覆

数据从源系统进数仓,要经过三个动作:**E**xtract(抽取)、**T**ransform(转换)、**L**oad(加载)。区别在 T 和 L 的顺序:

| | ETL(传统) | ELT(现代) |
|---|-----------|-----------|
| 顺序 | 抽取 → **先转换** → 再加载 | 抽取 → **先加载** → 再在数仓内转换 |
| 转换发生在 | 数仓**外部**的独立引擎(如 Informatica) | 数仓**内部**,用 SQL |
| 原始数据 | 转换后才入库,原始数据往往丢失 | 原始数据先完整入库,可随时回溯重算 |
| 算力 | 依赖专门的 ETL 服务器 | 借数仓自身的弹性算力(Trino/BigQuery/Snowflake) |
| 代表工具 | Informatica、SSIS | **dbt**(负责其中的 T) |

**为什么现代数仓转向 ELT**:云数仓算力变得便宜且弹性,「先把原始数据原样灌进去,再用 SQL 在库内转换」比「在外部先转好再灌」更灵活——原始数据留底,转换逻辑改了可随时重跑,而且转换就是普通 SQL,人人会写、易测试、易版本控制。

> **dbt 就是 ELT 里那个 "T"**。它假设你的原始数据已经在数仓里了(本项目用 `dbt seed` 模拟这一步),然后专注于用 SQL 把它转换成可用的数据。

## 为什么要分层

如果所有转换挤在一句巨大的 SQL 里,会变成没人敢碰的「意大利面」。数仓建模的核心思想是**分层**:每层只干一类事,层层递进,像流水线一样。

dbt 官方推荐的分层(也是本课程主线):

```
source(原始)→ staging(清洗)→ intermediate(中间)→ marts(业务)
```

- **source**:原始数据的声明入口。只声明,不转换。
- **staging**:一对一清洗每张源表(重命名、类型转换)。**不 JOIN、不聚合。**
- **intermediate**:承接复杂的、可复用的中间计算(需要 JOIN、多步聚合)。
- **marts**:面向业务问题的最终成品,供 BI/下游直接消费。

分层的好处:职责清晰、逻辑可复用、出问题好定位、每层可独立测试。

## Medallion 架构:另一套词汇,同一个思想

> **先定性**:本项目分层命名一律采用 dbt 官方 source/staging/intermediate/marts;Bronze/Silver/Gold 仅为帮你对照 Databricks 等其他资料,不是本项目采用的命名。

你在很多资料(尤其 Databricks 生态)会看到 **Medallion 架构**——Bronze / Silver / Gold 三层。它和 dbt 分层讲的是**同一件事**:

| Medallion | 含义 | 对应 dbt 分层 |
|-----------|------|--------------|
| 🥉 **Bronze** | 原始数据,原样落库 | source(+ seed) |
| 🥈 **Silver** | 清洗、去重、规范化、打平 | staging + intermediate |
| 🥇 **Gold** | 面向业务的聚合、可直接消费 | marts |

不用纠结用哪套词——理解「原始 → 清洗 → 业务」这个递进逻辑就够了。

## Kimball 维度建模:marts 层怎么组织

到了 marts(Gold)层,数据怎么摆才好用?业界主流是 **Kimball 的维度建模(dimensional modeling)**,核心是**星型模型(star schema)**,由两类表构成:

- **事实表(fact table)**:存业务过程的**度量/指标**——「发生了什么、多少」。如订单表:每行一笔交易,含金额、数量等可加总的数值。命名常用 `fct_` 前缀。
- **维度表(dimension table)**:存**描述性属性**——「谁、什么、何时、何地」。如客户表、产品表、日期表。命名常用 `dim_` 前缀。

事实表居中,维度表环绕四周,连线像星星,故名「星型」。查询时用事实表 JOIN 维度表,就能回答「上个月华东区 VIP 客户买了多少钱」这类业务问题。

> **本项目的做法**:采用 jaffle-shop 电商数据集,marts 层严格按星型模型拆分——两张事实表 `fct_orders`(订单粒度)、`fct_order_items`(订单行粒度,最细),三张维度表 `dim_customers` / `dim_products` / `dim_stores`。事实表存度量(金额、成本、毛利、商品数),维度表存描述属性,靠外键(`customer_id` / `store_id` / `product_sku`)相连。相比把维度和度量合在一张表的 OBT / One Big Table 风格,星型模型更贴近真实的大型数仓实践。本项目 `fct_order_items` 严格遵循此原则:描述属性(如产品名 `product_name`、类型 `product_type`)放 `dim_products`,事实表只留外键 + 度量,查询时按需 JOIN 取回——这是纯星型模型的做法。
