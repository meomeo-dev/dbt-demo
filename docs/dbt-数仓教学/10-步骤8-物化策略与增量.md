# 10 · 第 8 步:物化策略与增量上线

> 对应代码:`dbt_project.yml` 的 `+materialized: table`、`models/marts/fct_orders_incremental.sql`(incremental append 示例)
> 权威依据:dbt 官方文档 Materializations([docs.getdbt.com](https://docs.getdbt.com/docs/build/materializations))

## 物化(materialization)是什么

你写的 model 是一句 `select`。**物化决定 dbt 把这句 `select` 变成数仓里的什么东西** —— 一个视图?一张表?还是增量更新的表?这是性能与成本的核心权衡。

dbt 内置**五种**物化类型,默认是 **view**,通过 `materialized` 配置覆盖(可写在 `dbt_project.yml`、模型 `config()` 块或 YAML 属性文件里)。

## 五种物化类型对比

| 类型 | dbt 干什么 | 存储 | 查询速度 | 是否自动反映最新源数据 | 适用场景 |
|------|-----------|------|---------|----------------------|---------|
| **view**(默认) | 每次 run 执行 `create view as` | 不存数据 | 慢(每次查都重算) | ✅ 总是最新 | 轻量转换;**官方建议先用 view,遇到性能问题再换** |
| **table** | 每次 run 执行 `create table as`,重建整表 | 存全量 | 快 | ❌ 需重跑才更新 | 被 BI 工具频繁查询的模型 |
| **incremental** | 首次建表,之后只处理新增/变更行(merge/append) | 存全量 | 快 | 增量更新 | 大事实表、日志、事件流,全量重建太贵 |
| **ephemeral** | 不落库,作为 CTE 内联进下游模型 | 不存 | — | ✅ | 轻量中间逻辑,不想产生一堆中间表 |
| **materialized view** | 建数仓原生物化视图(需 adapter 支持) | 存 | 快 | 由数仓刷新 | 数仓支持时的自动增量 |

> 官方经验法则:**从 view 开始;查询慢了换 table;table 重建太贵、数据量大了换 incremental。**

## ⚠️ 本项目为什么全用 table

普通 dbt 项目 staging 层用 view 最省事。但本项目跑在 **Trino MySQL connector** 上,**它不支持 `CREATE VIEW`**,所以 view / 依赖 view 的 ephemeral 都用不了。`dbt_project.yml` 只能强制所有层用 table:

```yaml
models:
  dbt_demo:
    staging:
      +materialized: table    # Trino MySQL connector 不支持 CREATE VIEW
    intermediate:
      +materialized: table
    marts:
      +materialized: table
```

这不是最佳实践,而是**平台约束下的妥协**。换成 BigQuery/Snowflake 等数仓时,staging 应回归 view。

## incremental 增量模型(本项目 `fct_orders_incremental`,只能用 append)

当一张事实表有上亿行、每天只新增几十万行时,每次 `create table as` 全量重建既慢又贵。incremental 让 dbt **只处理新数据**。本项目的 `models/marts/fct_orders_incremental.sql` 就是一个可运行的增量模型:

```sql
{{
    config(
        materialized='incremental',
        incremental_strategy='append',   -- ⚠️ 本环境只能用 append,原因见下
        views_enabled=false              -- 强制增量的临时中间关系用 table 而非 view
    )
}}

with orders as (
    select * from {{ ref('stg_orders') }}
)

select
    order_id, customer_id, store_id, ordered_date,
    subtotal, tax_paid, order_total
from orders

{% if is_incremental() %}
  -- 仅在"增量运行"时执行:只追加比"已加载数据里最新日期"更新的订单
  where ordered_date > (select max(ordered_date) from {{ this }})
{% endif %}
```

- **`is_incremental()`**:一个宏。首次运行(表不存在)返回 false → 全量建表;之后返回 true → 只跑 `where` 过滤出的增量。
- **`{{ this }}`**:指代模型自己当前在库里的表。
- **`--full-refresh`**:`dbt run --full-refresh` 强制忽略增量逻辑、整表重建,用于修数据或改了表结构时。

## ⚠️ 平台约束(实测结论):本环境只能用 append

dbt 的 incremental 有多种 `incremental_strategy`,但**在 Trino MySQL connector 上只有 `append` 能跑**。这是本项目实测出来的硬结论:

| 策略 / 物化 | 结果 | 原因 |
|------------|------|------|
| `table` | ✅ 可用 | 纯 `CREATE TABLE AS` |
| `view` | ❌ 不支持 | connector 不支持 `CREATE VIEW`(全项目改 table) |
| incremental `append` | ✅ 可用 | 纯 `INSERT` 追加 |
| incremental `merge` | ❌ 报错 | `does not support MERGE with transactional execution` |
| incremental `delete+insert` | ❌ 报错 | Trino 的 `DELETE ... WHERE IN` 走 row-level MERGE,同样不支持 |
| `snapshot`(SCD2) | ❌ 首建成功、二次更新失败 | 依赖 MERGE 更新历史行 |

- **`append` 的局限**:只追加、无法更新或去重已存在的行,只适合「只增不改」的事件型数据(如订单流水)。会变更的业务表(需按 `unique_key` 更新)在本环境无法用 incremental 实现。
- **`merge` / `delete+insert` / `snapshot`** 都依赖事务性 MERGE,Trino MySQL connector 不支持,因此都跑不通。要用它们需换成 Hive / Iceberg / Delta 等支持事务 MERGE 的 connector。

## snapshots:记录缓慢变化维(SCD Type 2)

普通 table 只存「当前状态」。但业务上常需要「历史状态」——比如客户地址上个月是北京、这个月改成上海,你想两条都留档。这就是 **SCD Type 2**,dbt 用 **snapshots** 实现:

```sql
-- snapshots/ 目录下的示例(⚠️ 本环境跑不通,见下)
{% snapshot customers_snapshot %}
{{ config(unique_key='customer_id', strategy='timestamp', updated_at='updated_at') }}
select * from {{ source('raw', 'raw_customers') }}
{% endsnapshot %}
```

每次 `dbt snapshot`,dbt 会给变更的行「关闭旧版本、插入新版本」,自动维护 `dbt_valid_from` / `dbt_valid_to` 时间区间。

> **⚠️ 本项目实测**:snapshot 首次建表能成功,但第二次(更新历史行)会失败——因为它内部依赖事务性 MERGE,而 Trino MySQL connector 不支持。所以本项目没有可用的 snapshot,仅作概念讲解。真要用需换支持 MERGE 的 connector。

## 环境分离与调度(上线)

- **环境分离**:靠 `profiles.yml` 的多个 target(见 [第 02 章](02-连接配置-profiles.md))。开发用 `dbt run --target dev`(写到个人 schema),生产用 `--target prod`(写到生产 schema)。同一套代码,不同目标。
- **调度**:dbt 本身不带调度器。生产上线常见三种:
  - **dbt Cloud**:官方托管,内置调度 + CI/CD + 文档。
  - **Airflow**(常配 `astronomer-cosmos`):把 dbt DAG 编排进现有 Airflow 管道。
  - **cron**:最朴素,定时跑构建流程(本项目因 dbt-trino 时序坑用分步 `dbt seed && dbt run && dbt test`,见 [附录](附录-命令速查与常见坑.md))。
- **source freshness**:上线后可用 `dbt source freshness` 监控源数据是否按时到达(在 YAML 里配 `warn_after`/`error_after` 阈值 + `loaded_at_field`),数据延迟时告警。
