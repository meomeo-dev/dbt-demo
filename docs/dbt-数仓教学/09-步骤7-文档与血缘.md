# 09 · 第 7 步:文档与血缘 DAG

> 对应代码:全项目(`sources.yml` 的 description、各模型的 `ref()`/`source()`)

## dbt 文档:代码即文档

数仓最常见的痛点之一是「没人知道这张表哪来的、字段啥意思」。dbt 把文档和代码绑在一起:你在 YAML 里写的 `description`,加上模型间的 `ref()`/`source()` 关系,dbt 能自动生成一个**可交互的文档站点 + 血缘图(lineage graph / DAG)**。

## 描述从哪来

本项目的 `sources.yml` 里已经写了描述:

```yaml
sources:
  - name: raw
    description: "jaffle-shop 原始数据,由 dbt seed 从 CSV 载入 MySQL analytics schema"   # ← source 组描述
    tables:
      - name: raw_customers
        description: "原始客户表:每行一个客户"                        # ← 表描述
      - name: raw_orders
        description: "原始订单表:每行一笔订单(金额单位为分)"
```

模型的 description 本项目已提升到各层的 `schema.yml`(`stg_schema.yml` / `intermediate/schema.yml` / `marts/schema.yml`),给每个模型和关键列写了说明,`dbt docs` 会把它们收进文档站。SQL 文件顶部的注释是给读代码的人看的补充,真正进文档站的是 YAML 里的 `description` 字段。

## 血缘图(DAG)是怎么来的

你**从不手动画**血缘图。dbt 扫描所有 `ref()` 和 `source()`,自动推导出有向无环图(DAG):

```
source: raw.raw_customers ─▶ stg_customers ─────────────────────────▶ dim_customers
source: raw.raw_orders ────▶ stg_orders ──┬─────────────────────────▶ fct_orders
                                          ├─▶ int_orders_aggregated_to_customer ─▶ dim_customers
source: raw.raw_items ─────▶ stg_order_items ─┐
source: raw.raw_products ──▶ stg_products ────┼─▶ int_order_items_enriched ─┬─▶ fct_order_items
source: raw.raw_supplies ──▶ stg_supplies ─▶ int_supplies_aggregated_per_product ─▶ dim_products
                                          └─▶ int_order_items_aggregated_to_order ─▶ fct_orders
source: raw.raw_stores ────▶ stg_stores ────────────────────────────▶ dim_stores
```

> 注:`dim_customers` 引用 `int_orders_aggregated_to_customer`、`fct_orders` 引用 `int_order_items_aggregated_to_order`——改变粒度的聚合已下沉到 intermediate,marts 只做 JOIN 组装,`fct_orders` 不再依赖 `fct_order_items`。

- 每个 `{{ source('raw', 'raw_customers') }}` → 画一条「source → 模型」的边。
- 每个 `{{ ref('stg_customers') }}` → 画一条「模型 → 模型」的边。

这张图是 dbt 的核心资产:它既是**文档**(一眼看清数据从哪流到哪),也是**执行计划**(dbt 据此决定构建顺序、支持 `+model`/`model+` 这类图选择器)。

## 生成与预览

```bash
dbt docs generate     # 编译项目 + 抽取元数据,生成 target/catalog.json 等
dbt docs serve        # 起本地站点(默认 http://localhost:8080),含全文档 + 可点击血缘图
```

打开后你能:
- 浏览每个模型/source 的字段、描述、依赖、被依赖。
- 在 lineage 图上点任意节点,高亮它的上下游。
- 看到每个模型编译后的真实 SQL。

## 图选择器(靠血缘实现的高级用法)

血缘图让你能「按依赖关系」精准选择要跑的模型:

```bash
dbt run --select stg_customers+       # stg_customers 及其所有下游
dbt run --select +fct_orders          # fct_orders 及其所有上游
dbt run --select +fct_orders+         # fct_orders 及其全部上下游
```

改了 `stg_customers`?用 `stg_customers+` 就能只重跑受影响的部分,不用全量重来。
