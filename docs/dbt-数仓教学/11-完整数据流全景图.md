## dbt 官方分层 ↔ 本项目 ↔ Medallion(仅术语类比)

| dbt 官方分层 | Medallion(仅术语类比) | 本项目 | 产出 |
|-----------|-----------|--------|------|
| source(原始) | Bronze | seeds(jaffle-data 6 CSV)→ source `raw` | 6 张 `raw_*` 表 |
| staging(清洗) | Silver(部分) | `stg_*`(6 个) | 规范化的干净表 |
| intermediate(中间) | Silver | `int_supplies_aggregated_per_product` / `int_order_items_enriched` / `int_order_items_aggregated_to_order` / `int_orders_aggregated_to_customer`(4 个) | 聚合/打平的中间结果 |
| marts(业务) | Gold | `dim_*`(3)+ `fct_*`(2) | 星型模型:维度 + 事实 |

> Bronze/Silver/Gold 只是 Databricks 等资料里的术语类比,本项目命名一律用 dbt 官方 source/staging/intermediate/marts,详见 [第 00 章](00-数仓基础-ETL-vs-ELT.md)。marts 层的业务建模范式是 Kimball 星型建模,它与 dbt 工程分层是不同的轴,不要混称。

## 一条命令跑完整条链路

```bash
cd deploy && docker compose up -d && cd ..    # 起后端
dbt seed --full-refresh    # 第 1 步(6 张表)
dbt run                    # 第 3~5 步(dbt 靠 ref/source 自动排序:source→stg→int→mart)
dbt test                   # 第 6 步(67 个测试)
dbt docs generate && dbt docs serve   # 第 7 步
```

注意 `dbt run` 一条命令就跑完了 staging → intermediate → marts 全部 16 个模型 —— 你**从不手动指定顺序**,dbt 靠 `ref()` 建的血缘 DAG 自动决定构建次序(先 stg,再 int,最后 dim/fct)。这就是 dbt 相比手写 SQL 脚本最本质的价值。
