# 附录 · 命令速查与常见坑

> 对应代码:`CLAUDE.md`、`dbt_project.yml`

## 命令速查

```bash
# ── 启动后端(Trino + MySQL)──
cd deploy && docker compose up -d && cd ..
docker compose -f deploy/docker-compose.yml logs -f trino | grep "SERVER STARTED"  # 等这行出现

# ── 连通性检查 ──
dbt debug                       # 检查 profiles.yml + adapter + 能否连上 Trino

# ── 全量构建(顺序不可省)──
dbt seed --full-refresh         # 第 1 步:RAW 数据入库(6 张表)
dbt run                         # 第 3~5 步:staging → intermediate → marts(16 个模型)
dbt test                        # 第 6 步:质量校验(67 个测试)

# ── 精准操作 ──
dbt run  --select stg_customers        # 单个模型
dbt run  --select staging              # 整层(staging 6 个 / intermediate 4 个 / marts 5+1 个)
dbt run  --select +fct_orders          # 某模型 + 其全部上游
dbt test --select fct_orders           # 单模型的测试
dbt test --select source:raw           # 只跑 source 上的测试
dbt build --select stg_customers       # 单模型可用 build(run+test 一起)

# ── 改了 seed CSV 但表没更新 ──
dbt seed --full-refresh

# ── 文档 + 血缘图 ──
dbt docs generate && dbt docs serve
```

## dbt-trino 专属的 4 个坑(违反会报错或偶发失败)

这些是本项目区别于普通 dbt 教程的地方,是把标准 dbt 跑在 Trino→MySQL 上踩出来的。

### 坑 1:staging 必须用 `table`,不能用 `view`
Trino 的 MySQL connector 不支持 `CREATE VIEW`。普通 dbt 教程里 staging 默认 view,照抄会直接报错。本项目 `dbt_project.yml` 已对所有层设 `+materialized: table`。新增层沿用 table。→ 详见 [第 05 章](05-步骤3-staging清洗层.md)、[第 10 章](10-步骤8-物化策略与增量.md)。

### 坑 2:全量构建用分步 `seed → run → test`,别用 `dbt build`
`dbt build` 把 seed 和 source 测试编排进同一个并发 DAG。但 source 测试语义上不依赖 seed,会在 seed 建表的瞬间抢跑,触发 Trino 对 MySQL DDL 可见性的**瞬时竞态**,偶发 `TABLE_NOT_FOUND`。
- 全量:老实用 `dbt seed && dbt run && dbt test` 三步。
- 单模型:`dbt build --select x` 无此问题(没有 seed 与 source 测试的交叉)。

### 坑 3:JOIN 用显式 `ON`,不要用 `USING`
Trino 里 `LEFT JOIN ... USING (col)` 会把连接列合并成单一无别名列,后续 `c.col` / `o.col` 无法解析。一律写 `on c.col = o.col`。→ 详见 [第 07 章](07-步骤5-marts业务聚合层.md)。

### 坑 4:VirtioFS / 首次启动时序
首次启动 Trino 约需 30 秒,必须等到日志出现 `SERVER STARTED` 再跑 dbt,否则 `dbt debug` 连不上。

## 术语错位速记(最容易搞混)

| 你在 dbt 里写的 | 实际指的是 |
|----------------|-----------|
| `database: mysql` | **Trino catalog 名**(对应 `mysql.properties` 文件名),不是 "MySQL 数据库" |
| `schema: analytics` | MySQL 里真实的 `analytics` 库 |
| `type: trino` | dbt 只连 Trino,**从不直连 MySQL** |

## 版本约束

- **dbt Core 固定 1.11.x,不要升到 1.12**:VSCode Power User 插件(≤0.62.0)的 Python bridge 在序列化 dbt 1.12 返回对象时崩溃(`TypeError: must be real number, not _thread.lock`)。1.11.12 为规避版本。
- **dbt-trino 用 1.10.2**:兼容 core 1.10/1.11/1.12。
