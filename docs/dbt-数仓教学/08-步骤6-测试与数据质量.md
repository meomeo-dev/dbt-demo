# 08 · 第 6 步:测试与数据质量

> 对应代码:`models/staging/sources.yml`、各层 `schema.yml`(通用测试)、`tests/assert_order_total_positive.sql`(singular test)

## 为什么数仓一定要测试

转换写对了不代表数据是对的。上游脏数据、重复主键、意外的 NULL、非法枚举值 —— 这些不会让 SQL 报错,却会悄悄污染下游报表。dbt 的测试机制就是**在数据流动过程中自动断言「数据应该满足的规则」**,不满足就让 `dbt test` 失败,把问题挡在下游之前。

dbt 的测试本质很简单:**每个测试就是一句返回「违规行」的 `select`。返回 0 行 = 通过,返回 >0 行 = 失败。**

## 两类测试

### 1. 通用测试(generic tests,原 schema tests)
写在 YAML 里、可复用、参数化的测试。dbt 内置四个最常用的:

| 测试 | 断言 | 本项目用在哪 |
|------|------|-------------|
| `unique` | 该列无重复值 | 各表主键:`raw_customers.id`、`raw_orders.id`、`raw_products.sku`、`stg_*` 主键、`dim_*`/`fct_*` 主键 |
| `not_null` | 该列无 NULL | 上述主键 + 各外键列(`raw_orders.customer`、`raw_items.order_id` 等) |
| `accepted_values` | 该列只含指定枚举值 | `raw_products.type` 与 `stg_products.product_type` ∈ {jaffle, beverage} |
| `relationships` | 该列的值都能在另一张表找到(外键完整性) | `raw_orders.customer → raw_customers.id`、`raw_items.sku → raw_products.sku`、`fct_orders.customer_id → dim_customers` 等 |

本项目在 source 层就把测试直接挂在列上(`sources.yml`),清洗层与业务层的测试则写在各自的 `schema.yml` 里。看几个代表片段:

```yaml
# sources.yml —— source 层
tables:
  - name: raw_customers
    columns:
      - name: id
        tests: [unique, not_null]
  - name: raw_products
    columns:
      - name: type
        tests:
          - not_null
          - accepted_values:                    # 注意新语法:参数放进 arguments 块
              arguments:
                values: ['jaffle', 'beverage']
```

**`relationships`(外键完整性)本项目大量使用**。例如校验每笔订单的 `customer` 都存在于客户表,以及 marts 层事实表指向维度表:

```yaml
# sources.yml:raw_orders.customer → raw_customers.id
- name: customer
  tests:
    - not_null
    - relationships:
        to: source('raw', 'raw_customers')
        field: id

# marts/schema.yml:fct_orders.customer_id → dim_customers.customer_id
- name: customer_id
  tests:
    - not_null
    - relationships:
        to: ref('dim_customers')
        field: customer_id
```

### 2. 单点测试(singular tests)
当某条规则太特殊、无法用通用测试表达时,直接在 `tests/` 目录写一个 `.sql` 文件,里面是一句「查出违规行」的 SQL。本项目的 `tests/assert_order_total_positive.sql` 就校验「不允许出现金额为负的订单」:

```sql
-- tests/assert_order_total_positive.sql:订单总额不应为负
select
    order_id,
    order_total
from {{ ref('fct_orders') }}
where order_total < 0
```

返回任意行即判定失败(返回 0 行 = 通过)。

## 在哪层测什么(实践建议)

- **source 层**:测「数据刚进门」的基本健康度——主键唯一、外键非空、枚举合法。**越早发现上游问题越好。** 本项目就是这么做的。
- **staging 层**:测清洗后的字段(如转换后日期不为空)。
- **marts 层**:测业务规则(如聚合后金额非负、客户数与上游一致)。

## 扩展测试包

内置四个测试之外,社区两个包极常用(需在 `packages.yml` 声明后 `dbt deps` 安装):

- **`dbt_utils`**:补充大量通用测试,如 `unique_combination_of_columns`(联合唯一)、`accepted_range`(数值范围)、`not_null_proportion`(非空比例)等。
- **`dbt_expectations`**(移植自 Great Expectations):更丰富的数据质量断言,如列值分布、类型、正则匹配等。

> 本项目 demo 未引入这两个包,但真实项目做数据质量时几乎必装。

## 执行

```bash
dbt test                              # 跑全部测试(本项目共 67 个)
dbt test --select source:raw          # 只跑 source 上的测试
dbt test --select fct_orders          # 只跑某模型的测试
```

本项目全量 `dbt test` 预期 67 个测试全部通过(source 层 + staging/intermediate/marts 各 schema.yml + 1 个 singular test)。测试是 CI/CD 的核心关卡——PR 合并前跑一遍 `dbt run && dbt test`,红了就不许合。
